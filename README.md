# HR Tracker

MVP web app to track every MLB home run, day by day, and surface useful rolling
stats: HRs today, last 3 games, last 5 games, last 7 days, season total, and
last HR date.

**Stack**

- React 18 + Vite + TypeScript (frontend)
- Supabase Postgres (data store, RLS read-only for the browser)
- Node + TypeScript scripts via `tsx` (ingestion)
- Free/unofficial [MLB Stats API](https://statsapi.mlb.com/api/v1/) as the data source

This MVP intentionally skips: auth, paid subscriptions, betting odds.

## 3. Apply the database schema

Open the Supabase SQL editor and paste the contents of
[`supabase/schema.sql`](./supabase/schema.sql), then run it. It creates:

- `games` ŌĆö one row per MLB game, with `processed` flag, venue, probable
  pitchers (filled by the schedule importer)
- `home_runs` ŌĆö one row per HR event, deduped by `event_key`, with
  per-event matchup context (`batter_side`, `pitcher_throws`, `venue_name`)
- `player_daily_summary` ŌĆö rolling stats per `(player_id, date)`

Re-running the file is safe (idempotent).

**If you already have a populated DB**, apply the additive migrations in
order instead of re-running the full schema. They are idempotent:

1. [`supabase/migrations/001_matchup_context.sql`](./supabase/migrations/001_matchup_context.sql)
   ŌĆö adds handedness, venue_name, probable-pitcher columns + indexes.
2. [`supabase/migrations/002_venues.sql`](./supabase/migrations/002_venues.sql)
   ŌĆö adds `venue_id` to `home_runs` and `games`, plus the `venues`
   catalog and the `venue_hr_summary` derived cache.
3. [`supabase/migrations/003_players.sql`](./supabase/migrations/003_players.sql)
   ŌĆö adds the canonical `players` catalog so the frontend can resolve a
   hitter's *current MLB team* instead of falling back to whatever team
   string was on their last HR row (which can be a non-MLB name like
   "United States" if a WBC row was ever ingested).
4. [`supabase/migrations/004_weather_gametype.sql`](./supabase/migrations/004_weather_gametype.sql)
   ŌĆö adds `game_type` (used by `processDate` to skip non-MLB types) and
   `weather` placeholder columns (no fetcher wired yet; the HR Heat
   Score already reserves a `weather` slot with weight 0).
5. [`supabase/migrations/005_pitcher_starts.sql`](./supabase/migrations/005_pitcher_starts.sql)
   ŌĆö early shape of `pitcher_starts`. **Superseded by 006**; safe to skip
   if not already applied.
6. [`supabase/migrations/006_pitcher_starts_canonical.sql`](./supabase/migrations/006_pitcher_starts_canonical.sql)
   ŌĆö canonical `pitcher_starts` shape used by the code today: PK is
   `(game_id, pitcher_id)`, columns include `pitcher_hand`,
   `home_runs_allowed`, `innings_pitched`, `pitches`, `decision`, plus
   full `team/opponent/venue` denormalization. **Apply this one.** Ends
   with `NOTIFY pgrst, 'reload schema';` so PostgREST picks up the new
   columns immediately.
7. [`supabase/migrations/007_hr_target_snapshots.sql`](./supabase/migrations/007_hr_target_snapshots.sql)
   ŌĆö `hr_target_snapshots` table for the Backtest page. One row per
   (target_date, player_id, game_pk). Stores the persisted Top-N HR
   target ranking so we can join it against `home_runs` later to
   compute actual hit rates.
8. [`supabase/migrations/008_snapshot_type.sql`](./supabase/migrations/008_snapshot_type.sql)
   ŌĆö adds `snapshot_type text DEFAULT 'live'` (with a `CHECK in
   ('live', 'simulated')` constraint and an index). Distinguishes honest
   pre-game snapshots from historical backfills generated by
   `snapshot:range`.

After applying, backfill the new columns for existing rows:

```bash
npm run enrich:handedness   # fetches batSide / pitchHand from /v1/people/{id}
npm run enrich:venues       # fetches venue_name from each game's live feed
```

Both are safe to interrupt and re-run ŌĆö they only touch rows where the
target column is currently NULL.

## 4. Seed with one completed MLB date

```bash

```

This will:

1. fetch the schedule for that date and upsert into `games`
2. for every game whose status is "Final" and not yet `processed`, fetch its

3. rebuild `player_daily_summary` for that date

- **Betting research** ŌĆö Players with HR in back-to-back games, players with

- **Pitchers ŌĆö HR allowed** ŌĆö pitchers ranked by HRs allowed (season,
  L3 / L5 of their HR-allowed dates, L14d), with handedness when known
- **Handedness splits** ŌĆö league-wide HRs vs LHP / RHP, plus a per-player
  splits table (`vs LHP`, `vs RHP`, last-30-days versions). Player Detail

- **Ballparks** ŌĆö venue HR friendliness based purely on stored data

A separate **Today's Matchups** page (`/matchups`) surfaces, per scheduled
game on the as-of date: teams, venue, probable pitchers (or "TBD" if MLB
hasn't announced), each pitcher's recent HR-allowed stats, the top hot
hitters per team, and rule-based notes ("Hot hitter vs HR-prone pitcher",
"Lefty hitter vs RHP", "Power park recently").

The **HR Targets** page (`/targets`) is a **research-only ranking** (NOT
a guaranteed pick) of which hitters are most likely to homer in each
scheduled game. Date selector defaults to Today, with one-click Tomorrow
and a custom date picker. For each game it shows venue, probable pitchers
with handedness, and per-team ranked hitter tables with an "HR Heat
Score" column.

The Heat Score is a **weighted, normalized sum** (max 100). Each
component is normalized to 0..1 by dividing the raw stat by a
saturation cap, then multiplied by its weight:

| Component | Weight | Saturation cap | Source                                |
| --------- | -----: | -------------: | ------------------------------------- |
| **Season**| **40** | 30 HRs         | **structural base** ŌĆö season-to-date totals |
| Pitcher   | 15     |  6 HR allowed  | probable pitcher's L14d HR allowed    |
| L3 games  | 14     |  3 HRs         | player's last 3 distinct HR-dates     |
| Park      | 10     | 12 HRs         | venue's L14d HRs                      |
| Hand      | 10     |  1.0 share     | (HRs vs probable's hand) / known-hand |
| L5 games  |  8     |  4 HRs         | player's last 5 distinct HR-dates     |
| L7d       |  3     |  5 HRs         | calendar days [asOf-6, asOf]          |
| Weather   |  0     |  ŌĆö             | **placeholder** (no fetcher yet)      |

**Priority order** (as configured):

1. **SEASON POWER (base)** ŌĆö 40 pts. Season HR is the *structural base*,
   not just a component.
2. **RECENT FORM** ŌĆö L3+L5+L7d totals **25 pts**, dampened by stability.
   Recent form *boosts* the base; it does not replace it.
3. **PITCHER MATCHUP** ŌĆö 15 pts. HR-allowed history of the probable.
4. **HANDEDNESS** ŌĆö 10 pts. Batter side vs probable's hand.
5. **BALLPARK** ŌĆö 10 pts. Modifier only.

### Stability factor

Recent-form contributions are multiplied by:

```
effective = isElitePower ? max(season_hr, 15) : season_hr
stability = clamp(effective / 12, 0.35, 1.0)
```

|  Season HR | Stability (normal) | Stability (elite) |
| ---------: | -----------------: | ----------------: |
|  0 (cold)  |  0.35              | 1.00              |
|  5         |  0.42              | 1.00              |
|  8         |  0.67              | 1.00              |
| 12+        |  1.00              | 1.00              |

Pitcher / park / hand contributions are **not** dampened.

### Two hard guard rails

**1. Auto-elite at 12+ season HR.** Any player with Ōēź12 season HRs is
automatically treated as `isElitePower` for stability and Power Floor
purposes. The user no longer needs to add their name to the curated
list ŌĆö true power is detected from the data itself.

**2. Low-power cap.** A non-elite player with `season_hr < 5` AND
`hrs_l7d < 2` (calendar 7-day HRs) has their final heat **capped at 30**.
This prevents a single matchup edge (HR-prone pitcher ├Ś hot park) from
pushing a fringe hitter into the Top 10. A real 7-day streak (Ōēź2 HRs in
the past week) exempts them.

The cap uses `hrs_l7d` rather than `hrs_l3` because `hrs_l3` is
"player's most recent 3 HR-dates" ŌĆö for a fringe hitter who last
homered 2 months ago, that metric can read as 2 even though they have
no current heat. Calendar-recent HRs are the correct exemption signal.

### Power Floor ŌĆö solving "Aaron Judge ranked too low"

The model only sees *current-season* HRs. Early in a season, an elite
slugger has accumulated few HRs and would score low purely on count.
The **Power Floor** lifts known elites so a slow-start Judge isn't
buried below a streak from a non-elite hitter:

- Hardcoded `ELITE_POWER_NAMES` set in `src/lib/stats.ts` (Judge, Ohtani,
  Schwarber, Alonso, Olson, Seager, Soto, Alvarez, Betts, Harper, etc.).
- Frontend matches the canonical `players.full_name` (case-insensitive)
  against this set to derive `elitePowerIds`.
- For elite players: `season_power` normalized = `max(actual, 0.7)`
  (Ōēł21 / 30 pts), and stability uses `max(season_hr, 15)`.

The expandable detail surfaces this with a star (Ōśģ) next to the player
name and a `"Ōśģ elite (power floor)"` annotation on the Season-power tile.

#### Market Sanity Mode (future hook)

A `DailyHrOdds` TS interface is declared as a future hook for ingesting
sportsbook HR-prop lines. When wired, odds will override the curated
Power Floor for any player whose implied probability is above threshold.
Until then, the curated elite list is the safety net so high-market
sluggers can never disappear from the page.

#### "Model disagreement" warning

When any elite-flagged player ranks worse than #30 across all boards,
the page surfaces an orange-bordered warning panel listing them above
the Top 10. That's the safety net for "Judge is buried at #45" ŌĆö
you'll see the disagreement explicitly with their season total, current
heat, and reason strings.

### Expandable score breakdown

Click any row in the Top 10 HR Targets table to expand a detail panel.
The first thing shown is the grouped breakdown ŌĆö six tiles:

```
Season power (base) ┬Ę Recent form (├Ś stability) ┬Ę Pitcher ┬Ę Handedness ┬Ę Venue ┬Ę Final heat
```

Below that, an explicit **raw ŌåÆ final** explanation showing every
adjustment that the model applied:

```
Raw score: 22.0  ŌåÆ  Final: 49.0
 - Power Floor (12+ season HR, auto)  (+27.0)
```

Or, for a capped player:

```
Raw score: 35.4  ŌåÆ  Final: 30.0
 - Low-power cap (Ōēż30; <5 season HR + <2 HR L7d)  (-5.4)
```

Then: per-component contribution bars (Season, L3, L5, L7d, Pitcher,
Park, Hand) and the probable pitcher's HR-allowed line, venue rank,
and batter splits.

```
heat_score = ╬Ż (weight_i * clamp(raw_i / saturation_i, 0, 1))
           Ōēż 100
```

Weights live in `HEAT_SCORE_WEIGHTS` (in `src/lib/stats.ts`). To tune
the model, edit one object ŌĆö no math changes needed. The weights are
asserted to sum to 100 in the fixture so an accidental edit doesn't
silently shift the scale.

**Why this shape?** The previous formula was uncapped on recent form,
so a 4-HR-in-3-games player could swamp every other signal. With caps
and weights, L3 + L5 still total 55% of the score but no single
component can dominate to the exclusion of others.

Each component degrades gracefully when its inputs are missing ŌĆö a game
with no probable pitcher still ranks hitters by recent form, season
power, and park boost. Reasons displayed next to each row are **specific
and numeric** ŌĆö up to 4 per player, prioritized by contribution to the
score:

- `3 HR in last 2 games` (L2 streak wins over L3 when present)
- `2 HR in last 3 games` / `4 HR in last 5 games` / `3 HR in last 7 days`
- `HR in 3 straight games` (consecutive-calendar-day streak)
- `Elite power profile (32 season HR)` (when season HR Ōēź 25 ŌĆö surfaces alongside the count)
- `28 season HR` (when 15 Ōēż season HR < 25; mid-power players still get the tag when nothing stronger)
- `5 HR vs RHP this season` (specific count vs the probable's hand) or `Strong vs RHP` when ratio is high on small samples
- `Pitcher allowed 6 HR in last 3 starts` (preferred) or `Pitcher allowed 5 HR in last 14 days`
- `Venue top 3 HR park in L14d` (rank-based, top 5) or `Hitter-friendly venue (4 HR L14d)`
- **Pitcher narrative tags** when starts data shows real trouble:
  `RHP allowing elevated HR rate` / `LHP allowing elevated HR rate` /
  `Weak HR suppression recently`
- **Meta:** `Hot streak + favorable matchup` when L3+L5 + (pitcher OR park OR hand) all score above 60-70% of their weight thresholds

The page shows a **Top 10 overall HR Targets** table at the top
(flattened across every game on the date, sorted by Heat Score
descending), then per-game cards below ŌĆö also sorted by Heat Score
within each card. A prominent disclaimer at the top reinforces that

### Backtest page (`/backtest`)

Persists daily Top-N HR target rankings into `hr_target_snapshots` and
joins them against actual HRs from `home_runs` to compute hit rates.

```bash
# Manually snapshot a date (skipped if rows already exist; --force overwrites)
npm run snapshot:targets -- 2026-05-10
npm run snapshot:targets -- today
npm run snapshot:targets -- yesterday
npm run snapshot:targets -- 2026-05-10 --force --limit 25
```

The orchestrator does this automatically:

- `update:daily` / `update:morning` ŌåÆ snapshot today's targets
- `update:night` ŌåÆ snapshot today's targets AND tomorrow's targets (since
  tomorrow's schedule + probables are freshly refreshed by then)

The page shows:

- KPI strip: target date, snapshot size, HRs on date, top 10 hit rate
- Hit-rate cells for Top 3 / Top 5 / Top 10 / All
- Top 10 table with rank, player, team, opponent, heat score, **Ō£ō/Ō£Ś**,
  number of HRs hit that day, and the reason string from the snapshot
- Ranks 11+ in a second table for full context

Server-side snapshot logic uses the exact same model as the HR Targets
page (imports from `src/lib/stats.ts`), so the persisted ranking
matches what the page would have shown when it was generated.

**No `players` table required.** The snapshot script derives the player
index (name + team per player_id) directly from `home_runs` by taking
the most-recent observation per player. Only `home_runs` and `games`
are required for snapshotting; `pitcher_starts` is used when present
for richer pitcher form. So if you haven't applied the (optional)
canonical `players` migration, snapshots still work ŌĆö they fall back
to the per-HR team string for each player, which matches the schedule
API's team names for regular-season rows.

**Filters**: **as of date**, **team**, **player search**. Plus quick controls
to step the date one day at a time (ŌŚĆ Day / Today / Day Ō¢Č). Mobile-responsive
(filters stack, tables scroll horizontally on narrow screens).

**Mental model**: "Last 3 games" means each player's most-recent 3 HR-game
dates *up to and including* the as-of date ŌĆö pulled from the full stored
history, NOT from a tiny windowed query. So if you select 2026-05-09:

- Last 3 games ŌåÆ each player's last 3 HR dates Ōēż 2026-05-09
- Last 5 games ŌåÆ each player's last 5 HR dates Ōēż 2026-05-09
- Last 7 days ŌåÆ 2026-05-03 through 2026-05-09
- Last 14 days ŌåÆ 2026-04-26 through 2026-05-09
- Season total ŌåÆ all HRs in 2026 from Jan 1 through 2026-05-09

Player detail page is unchanged ŌĆö click any player name in any panel to see
their full HR log and rolling stats.

If the dashboard is empty, run a backfill (see above). All raw HRs stay in
`home_runs` forever ŌĆö old data is never deleted.

### Long-term prediction workflow

The app has four named phases in a daily cycle. Running `npm run update:daily`
performs all of them in order, but they each have a distinct purpose worth
knowing:

#### 1. Morning update (~8 AM ET)

Run `npm run update:daily`. This:

- Pulls **schedules** for yesterday, today, tomorrow, and today + 3 days
  (`enrich:schedule`).
- **Processes completed games** from yesterday ŌĆö extracts HRs into
  `home_runs`, captures every starter's line into `pitcher_starts`.
- Runs all **enrichments** (probable pitchers via live-feed fallback,
  handedness, venues, pitcher-starts backfill, players catalog).
- Rebuilds **player_daily_summary** for yesterday + today.

#### 2. Pre-game snapshot (also in morning update)

The morning run also takes a **Saved Snapshot** of HR Targets for **today**
and **tomorrow**, with skip-if-exists semantics. The first snapshot of the
day is the canonical pre-game record. Later same-day runs see it already
exists and leave it alone ŌĆö so Backtest always has an honest pre-game
ranking to compare against actual results.

Each snapshot row carries a `snapshot_type` that auto-detects to `live`
when taken before first pitch on a future/today date, or `simulated` when
backfilled for a past date.

```bash
npm run snapshot:today              # explicit shortcut (also skip-if-exists)
npm run snapshot:tomorrow
npm run snapshot:targets -- 2026-05-12 --force   # deliberate overwrite
```

#### 3. Night results processing (~10 PM ET)

Run `npm run update:daily` again. Now today's finalized games get
processed ŌĆö their HRs land in `home_runs` and starters in `pitcher_starts`.
The snapshot phase still skips-if-exists, so the morning's saved ranking
is preserved untouched.

#### 4. Backtest comparison (next morning)

Visit `/backtest?date=YESTERDAY`. The page reads the Saved Snapshot from
yesterday's morning run and joins it against `home_runs` to compute Top
3 / 5 / 10 hit rates. Because the snapshot was locked in pre-game, the
comparison is honest ŌĆö no hindsight bias.

To make sure `/targets` and `/backtest` show the **same Top 10** for that
date, HR Targets defaults to Saved Snapshot view whenever one exists.
You can toggle to Live Preview if you want to see the current model
output (with a clear "may differ from Backtest" warning), or to Compare
mode to see the rank drift side-by-side.

### `update:daily` orchestrator

The intended cadence is one command for all daily upkeep:

```bash
# Local ŌĆö run once each morning. Idempotent; safe to re-run anytime.
npm run update:daily
```

This single command does the entire daily upkeep: schedules, completed-game
ingest, all enrichments, **pre-game HR Target snapshots for today + tomorrow**,
and summary rebuild.

For automated runs, the included GitHub Actions workflow runs `update:daily`
on a four-tick cadence (8 AM, 12 PM, 3 PM, 10 PM ET ŌĆö see the cron table
below). Each tick is small, idempotent, and skips work already done. The
snapshot step is **skip-if-exists by default** so the earliest run's
predictions are preserved as the canonical pre-game record; later runs
won't overwrite them unless you explicitly pass `--force`.

```bash
# Quick snapshot shortcuts (skip-if-exists by default)
npm run snapshot:today
npm run snapshot:tomorrow

# Manual / forced regeneration for a specific date
npm run snapshot:targets -- 2026-05-11 --force

# Historical backfill for a date range (auto-tagged 'simulated')
npm run snapshot:range -- --start 2026-05-01 --end 2026-05-11
npm run snapshot:range -- --start 2026-05-01 --end 2026-05-11 --force
npm run snapshot:range -- --start 2026-05-01 --end 2026-05-11 --limit 25 --delay 200
```

#### Snapshot types: `live` vs `simulated`

Every row in `hr_target_snapshots` carries a `snapshot_type` that tells
the Backtest page how to interpret it:

- **`live`** ŌĆö the snapshot was taken **before** any game on its
  target_date had started. This is the honest pre-game ranking
  `update:daily` aims for.
- **`simulated`** ŌĆö the snapshot was taken **after** some games on its
  target_date had already played (either same-day post-start or as a
  historical backfill via `snapshot:range`). The math is the same but
  the inputs are not pristine pre-game, so the rows are flagged so you
  don't mistake them for honest predictions.

Auto-detection rules:

| Scenario | snapshot_type |
| --- | --- |
| target_date is in the past | `simulated` |
| target_date is today, no games started yet | `live` |
| target_date is today, Ōēź1 game already in progress / final | `simulated` |
| target_date is in the future, schedule announced | `live` |

The Backtest page shows a coloured badge ŌĆö green `ŌŚÅ Live pre-game` or
orange `ŌŚŗ Simulated historical` ŌĆö next to the snapshot timestamp.

### `update:daily` pipeline

`npm run update:daily` runs the entire pipeline in a safe, idempotent order:

1. **Schedules** ŌĆö `enrich:schedule` for `[yesterday, today+3]`. Null-safe:
   if MLB returns `null` for a probable pitcher this run, the existing
   value is preserved (we never erase good data with TBD).
2. **Process completed games** ŌĆö `processDate(yesterday)` and/or
   `processDate(today)`. Only games whose status is `Final` and aren't
   already marked `processed` get HR + pitcher-start extraction.
3. **Enrich data** ŌĆö `enrich:probable-pitchers` (live-feed fallback),
   `enrich:handedness` (cap 100/run), `enrich:venues` (cap 30/run +
   `venue_hr_summary` rebuild), `enrich:pitcher-starts` (last 14 days),
   `enrich:players` (cap 100/run, only when Ōēź7 days stale).
4. **Snapshot HR Targets** ŌĆö persists the Top-N ranking for **today and
   tomorrow** into `hr_target_snapshots`. Skip-if-exists semantics:
   the first run of the day creates the snapshot, later same-day runs
   leave it alone. Warns "Snapshot may not be clean pre-game" if any
   game on the target date has already started by the time the snapshot
   runs (logs only; snapshot still proceeds).
5. **Rebuild summaries** ŌĆö `rebuildPlayerSummaries` for yesterday and
   today so the `player_daily_summary` cache stays current.

Every step is **isolated**: a failure in one doesn't kill the run, and
re-running is always safe. No data is ever deleted. HR rows dedup by
`event_key`. Game rows upsert with null-safe semantics. The full cache
wipe lives in `npm run rebuild:all` and is never invoked from here.

### Enrichment pipeline

The MVP captures everything an event-level live feed gives us during
`process:date`. For *historical* rows that predate a column being added,
or for *upcoming* games whose probable pitchers MLB hasn't announced yet,
run the enrichment scripts:

```bash
# 1. Apply the migrations to your existing DB (idempotent):
#    paste supabase/migrations/001_matchup_context.sql then 002_venues.sql
#    into the Supabase SQL editor.

# 2. Backfill batter_side & pitcher_throws on every existing HR row.
npm run enrich:handedness
# Useful flags:
#   --dry-run                # report only, no writes
#   --limit 100              # cap people lookups this run
#   --pitchers-only / --batters-only
#   --delay 500              # ms between API calls (default 200)

# 3. Backfill venue_id + venue_name on every existing HR + game row, refresh
#    the canonical venues catalog, and rebuild the venue_hr_summary derived cache.
npm run enrich:venues
# Useful flags:
#   --dry-run                # report only
#   --limit 50               # cap games per run (good for first try)
#   --skip-summary           # don't rebuild venue_hr_summary
#   --as-of 2026-05-09       # anchor for venue_hr_summary (default: today)

# 4. (Optional) Refresh probable pitchers + venue for upcoming dates without
#    touching home_runs. Run this 1-2x per day on a cron.
npm run enrich:schedule
# Defaults to today..today+7. Override with:
#   --start 2026-05-10 --end 2026-05-17
#   --dry-run

# 5. Populate / refresh the canonical players catalog so the UI can show each
#    player's *current MLB team* instead of the per-HR team (which may be
#    "United States" or similar from a WBC / exhibition row). Run after a
#    backfill, then weekly to keep up with mid-season trades.
npm run enrich:players
# Useful flags:
#   --dry-run                # report only
#   --limit 200              # cap people lookups this run
#   --refresh-days 3         # re-fetch any players row older than 3 days (default 7)
#   --force                  # ignore freshness check, refresh every player

# 6. Fallback: fill probable pitchers that the schedule API didn't return,
#    using each game's live feed (gameData.probablePitchers + actual starter
#    from the boxscore as the deepest fallback). Resolves handedness inline.
#    Run this if the HR Targets page shows "TBD" pitchers despite enrich:schedule.
npm run enrich:probable-pitchers
# Useful flags:
#   --start 2026-05-01 --end 2026-05-15
#   --dry-run
#   --limit 50
```

All three scripts are **idempotent** and **safe to interrupt and resume**:

- `enrich:handedness` only touches rows where `batter_side` (or
  `pitcher_throws`) is currently `NULL`. Re-running picks up where it left off.
- `enrich:venues` only touches rows where `venue_id` or `venue_name` is
  currently `NULL`. The summary rebuild at the end is a full recompute from
  raw `home_runs`, so the cache is always in sync with the source of truth.
- `enrich:schedule` overwrites probable-pitcher fields on `games` (this is
  intentional ŌĆö those values change as MLB updates them). It never modifies
  `processed`, `processed_at`, or any HR rows.
- `enrich:players` skips any player row updated within `--refresh-days` (default 7)
  unless `--force` is passed. So daily re-runs are cheap.

All four scripts retry transient HTTP errors (5xx / network blips) automatically.

### Why "Aaron Judge ŌĆö United States" can no longer happen

If a non-MLB game (WBC, exhibition) was ever ingested, the `home_runs.team`
column can hold a country name like "United States." The frontend used to
read team straight from the most recent HR row, so player pages and
leaderboards inherited that bad string.

Now:
- `npm run enrich:players` materializes each player's *current MLB team*
  from `/v1/people/{id}` (the `currentTeam.name` field) into a canonical
  `players` table.
- On every dashboard load, the frontend fetches the `players` table and
  uses `applyCanonicalTeams()` to remap each HR row's `team` to the
  canonical value before any aggregation, leaderboard, or display.
- Players not yet in the table pass through with their per-HR team
  unchanged ŌĆö graceful degradation, never a crash.

This is enforced by a regression assertion in the stats fixture
(`team leaderboard does not contain United States`).

### Source-of-truth model

`home_runs` is the **single source of truth**. Every other table (and the
entire frontend) derives from it:

- The Dashboard fetches all HRs from Jan 1 of the as-of year through the
  selected as-of date and computes every leaderboard, hot-hitter view, team
  total, back-to-back streak, and 2+ HR pattern from that raw set.
- The Player Detail page fetches every HR row for the player and computes
  season total, last HR date, L3/L5 games, L7d/L14d entirely from those
  rows. Both panes use the same `computePlayerView` / aggregation helpers
  in `src/lib/stats.ts`, so they can never disagree.
- `player_daily_summary` is a **derived cache** maintained for future
  server-side consumers (Slack bots, scheduled digests). The frontend does
  NOT read from it. If you ever suspect drift, run `npm run rebuild:all`
  to wipe and recompute it from `home_runs`.

If you ever see numbers that don't match between the dashboard and the
player page, it's a bug in `stats.ts` ŌĆö not a stale-cache problem ŌĆö because
both pages share that math.

### Backfilling a full season

```bash
# Ingest every date from 2026-03-01 through 2026-05-09
npm run backfill:season -- 2026-03-01 2026-05-09

# Slower / more polite (500ms between dates instead of the default 350ms)
npm run backfill:season -- 2026-03-01 2026-05-09 --delay 500

# Also rebuild player summaries after each date (slower; usually unnecessary
# because the dashboard now computes rolling stats client-side)
npm run backfill:season -- 2026-03-01 2026-05-09 --rebuild-per-day
```

The backfill is **safe to interrupt and resume** ŌĆö re-running it over the same
range only touches games that aren't yet `processed=true`, and HR rows are
deduped by `event_key`. Old data is never deleted.

`process:date` accepts: a YYYY-MM-DD string, the literal `today`, or `yesterday`.

## Recommended ingest cadence

The MVP is designed to **not over-poll** the public MLB API. A reasonable
production cadence:

1. **Morning run** ŌĆö process yesterday's now-finalized games:
   `npm run process:yesterday`
2. **Late-night run** ŌĆö process today's completed games:
   `npm run process:today`
3. *(optional, later)* a light poll every 15ŌĆō30 min during game windows, only
   touching games whose status is "In Progress" or "Final" and `processed=false`.

`processDate` and `rebuildPlayerSummaries` are both **safe to re-run** ŌĆö
games marked `processed=true` are skipped and HRs are deduped by `event_key`.

### Wiring it up to cron later

Once deployed somewhere with cron (Railway, Render, Fly, a small VPS, GitHub
Actions), add jobs like:

```cron
# 10:00 UTC: yesterday's games are guaranteed finalized
0 10 * * *  cd /app && npm run process:yesterday >> logs/cron.log 2>&1

# 06:00 UTC: late west-coast games from "today"
0 6 * * *   cd /app && npm run process:today >> logs/cron.log 2>&1
```

Or, for GitHub Actions, schedule a workflow that runs the same npm scripts
with the env vars provided as repo secrets.

## How home runs are detected

In `extractHomeRuns.ts` we walk `liveData.plays.allPlays[]` from the
`/v1.1/game/{gamePk}/feed/live` response and keep plays where:

```ts
play.result.eventType === 'home_run'   // or play.result.event === 'Home Run'
```

For each, we read `matchup.batter`, `matchup.pitcher`, `about.inning`, and the
last `playEvents[*].hitData` (Statcast ŌĆö exit velocity, launch angle, total
distance) when present. The dedup key is
`${gamePk}-${batterId}-${atBatIndex}`, which is unique per HR event in a game.
